### DAY 10 — "NETWORKING — HOW DATA ACTUALLY TRAVELS"
**Scene:** Every request your application handles is a packet traveling through layers.
Today you learn those layers — because interviewers love asking about them.

```
0:00 – 0:10  WATCH: ByteByteGo — "How does the internet work?" (YouTube, ~10 mins)
             Take 3 notes as you watch.

0:10 – 0:40  LEARN THE OSI MODEL (just enough, not all 7 layers in depth):

             The ones that matter for SRE interviews:

             Layer 4 — Transport: TCP vs UDP
             → TCP: reliable, ordered, connection-based (3-way handshake)
                    Used for: HTTP, HTTPS, SSH, databases
             → UDP: fast, no guarantee, connectionless
                    Used for: DNS, video streaming, gaming

             Layer 7 — Application: HTTP/HTTPS, DNS, SSH
             → This is where your apps live

             The 3-way handshake (TCP connection):
             Client → Server: SYN ("I want to connect")
             Server → Client: SYN-ACK ("OK, I'm ready")
             Client → Server: ACK ("Great, let's talk")
             → After this, data flows

             Interview question: "What's the difference between TCP and UDP?"
             Answer: TCP guarantees delivery and order via the 3-way handshake.
             UDP is faster but doesn't guarantee delivery. Use TCP for anything
             where losing data is unacceptable (APIs, databases). Use UDP for
             things where speed matters more than reliability (DNS lookups, video).

### DAY 11 — "DNS — THE INTERNET'S PHONE BOOK"
**Scene:** When you type google.com, something translates that to an IP address.
That something is DNS. Every SRE must understand it cold.

```
0:00 – 0:10  WATCH: ByteByteGo — "How DNS Works" (YouTube, ~8 mins)
             Pause and replay any part you don't understand.

0:10 – 0:40  UNDERSTAND DNS RESOLUTION STEP BY STEP:

             When you type "google.com":
             Step 1: Browser checks its own cache. Got it? Done. Skip all steps.
             Step 2: OS checks /etc/hosts file. Got it? Done.
             Step 3: OS asks the Recursive Resolver (usually your ISP or 8.8.8.8)
             Step 4: Recursive Resolver checks its cache. Got it? Returns it.
             Step 5: Recursive Resolver asks Root Name Server → "who handles .com?"
             Step 6: Root says → "ask .com TLD server"
             Step 7: TLD server says → "ask google.com's authoritative server"
             Step 8: Authoritative server returns the IP address
             Step 9: Recursive Resolver caches it (TTL) and returns it to you
             Step 10: Browser connects to that IP

             Draw this flow on paper. Seriously. Draw it.

0:40 – 1:00  PRACTICE DNS COMMANDS:
             $ nslookup google.com          → basic DNS lookup
             $ dig google.com               → detailed DNS info
             $ dig google.com +trace        → watch every step of resolution
             $ cat /etc/hosts               → your local DNS override file
             $ cat /etc/resolv.conf         → which DNS server your system uses

             DNS Record Types to know:
             A     → domain → IPv4 address (most common)
             AAAA  → domain → IPv6 address
             CNAME → alias → another domain (e.g., www → example.com)
             MX    → mail server for a domain
             TXT   → text info (used for SPF, verification)
             NS    → nameserver for the domain

1:00 – 1:30  RUBBER DUCK:
             Write: "When a user types myapp.com, here is exactly what happens..."
             (Write every step from memory. Check your notes. Fill gaps.)
             Write: "TTL in DNS means..."
             Write: "A CNAME record is different from an A record because..."
```

### DAY 12 — "LOAD BALANCERS + HTTP"
**Scene:** Every app you've run sits behind a load balancer. Today you explain exactly what it does.

```
0:00 – 0:15  UNDERSTAND WHAT A LOAD BALANCER DOES:

             Problem: You have 3 servers. How does traffic get distributed?
             Solution: Load balancer sits in front and decides.

             Types of load balancing algorithms:
             Round Robin    → request 1 → server 1, request 2 → server 2, request 3 → server 3...
             Least Connections → send to server with fewest active connections
             IP Hash        → same client always goes to same server (session stickiness)

             AWS ALB (Application Load Balancer) — Layer 7:
             → Understands HTTP/HTTPS
             → Routes based on URL path (/api → service A, /web → service B)
             → Does health checks (if server unhealthy → stop sending traffic)
             → Terminates SSL (decrypts HTTPS, forwards HTTP to backends)

             AWS NLB (Network Load Balancer) — Layer 4:
             → Understands TCP/UDP only
             → Faster, but no HTTP-level routing
             → Used for non-HTTP traffic (databases, gaming, etc.)

0:15 – 0:40  UNDERSTAND HTTP:

             HTTP Methods (must know cold):
             GET    → retrieve data (no body)
             POST   → create data (has body)
             PUT    → replace entire resource
             PATCH  → update part of resource
             DELETE → delete resource

             HTTP Status Codes (must know cold):
             2xx = Success
               200 OK, 201 Created, 204 No Content
             3xx = Redirect
               301 Moved Permanently, 302 Found (temporary)
             4xx = Client Error (YOU did something wrong)
               400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
             5xx = Server Error (SERVER did something wrong)
               500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable
               504 Gateway Timeout

             For SREs: 502 = your upstream server is down or returned garbage
                        503 = your server is overloaded or in maintenance
                        504 = your server responded too slowly (timeout)

0:40 – 1:00  PRACTICE:
             $ curl -I https://google.com      → see HTTP headers + status code
             $ curl -v https://google.com      → verbose, see full request/response
             $ curl -w "%{http_code}" https://yourapp.com  → just the status code

1:00 – 1:30  RUBBER DUCK:
             Write: "The difference between ALB and NLB is..."
             Write: "When I see a 502 error, it means... and I would debug it by..."
             Write: "Round robin load balancing works like..."
```

**Deliverable:** HTTP status codes memorized (test yourself), rubber duck paragraphs done.

0:10 – 0:50  Write your answer — every step, in order:

             Step 1: Browser checks DNS cache for myapp.com
             Step 2: OS checks /etc/hosts
             Step 3: DNS recursive resolver contacted → root → TLD → authoritative
             Step 4: Returns IP address of load balancer (e.g., 52.12.34.56)
             Step 5: Browser initiates TCP 3-way handshake with that IP on port 443
             Step 6: TLS handshake begins — negotiate cipher, exchange certificates

             Step 8: Browser sends HTTP GET / request
             Step 9: ALB receives request, checks routing rules
             Step 10: ALB does health check on backend — picks healthy server
             Step 11: ALB forwards request to backend server (your app) on port 80
             Step 12: App processes request, hits database if needed

### DAY 20 — "K8s RESOURCE MANAGEMENT"
**Scene:** Your cluster runs out of CPU and pods start failing. How do you prevent this?
Resource requests, limits, and HPA are your tools.

```
0:00 – 0:10  THE CORE PROBLEM:
             If pods can use unlimited CPU/memory → one buggy app kills the whole node.
             Solution: requests (guaranteed) and limits (maximum).

0:10 – 0:45  UNDERSTAND REQUESTS VS LIMITS:

             Requests: "I need at LEAST this much CPU/memory"
             → Used by scheduler to pick which node can fit this pod
             → If node has 4 CPU and pod requests 2 CPU → scheduler assigns it there

             Limits: "I can use AT MOST this much CPU/memory"
             → If pod exceeds CPU limit → CPU throttled (slows down)
             → If pod exceeds memory limit → OOMKilled (process killed immediately)

             Best practice:
             → Set requests = what your app normally uses
             → Set limits = 2x requests (some headroom)
             → NEVER set limits without setting requests

             Read this YAML:
             resources:
               requests:
                 memory: "256Mi"
                 cpu: "250m"       # 250 millicores = 0.25 CPU
               limits:
                 memory: "512Mi"
                 cpu: "500m"

0:45 – 1:05  UNDERSTAND HPA (Horizontal Pod Autoscaler):
             When CPU usage > 70% → HPA adds more pod replicas automatically
             When CPU usage drops → HPA removes extra replicas

             $ kubectl get hpa               → see autoscalers
             $ kubectl describe hpa [name]   → see current metrics and targets

             Interview question: "How does auto-scaling work in your Kubernetes setup?"
             Answer: "I use HPA which monitors CPU/memory metrics via metrics-server.
             When average CPU across pods exceeds the target threshold (usually 70%),
             HPA scales up the deployment. Scale-down has a cooldown period to
             prevent flapping. For node-level scaling, I use Cluster Autoscaler
             which adds EC2 nodes when pods can't be scheduled."

### DAY 24 — "AWS RDS + ELASTICACHE — DATABASE SRE"
**Scene:** Databases are the hardest part of reliability. Know them cold.

```
0:00 – 0:50  UNDERSTAND RDS HIGH AVAILABILITY:

             Redis is cache — it can be lost. Never store source-of-truth data in cache.

### DAY 31 — "K8s NETWORKING DEEP DIVE — CNI"
```
0:00 – 0:10  THE PROBLEM CNI SOLVES:
             Pods need to communicate across nodes.
             Each pod gets its own IP. How does pod-A on node-1 reach pod-B on node-2?
             Answer: CNI (Container Network Interface) plugin handles this.

0:10 – 0:50  UNDERSTAND CNI PLUGINS:

             Flannel: Simple overlay network using VXLAN tunneling
             → Good for: simple setups, learning
             → Bad for: performance-critical production (overhead from encapsulation)

             Calico: Layer 3 networking using BGP routing
             → Good for: production, better performance, supports NetworkPolicies well
             → Used in many enterprise K8s deployments

             AWS VPC CNI (for EKS):
             → Pods get REAL VPC IP addresses (not overlay)
             → Pods are directly addressable within VPC
             → Better integration with AWS security groups
             → This is what YOUR EKS cluster uses

             How pod-to-pod communication works (EKS with VPC CNI):
             Pod-A (node-1) → sends packet to Pod-B IP
             → AWS VPC routing table knows Pod-B IP → routes to node-2
             → node-2's kubelet receives → delivers to Pod-B
             No encapsulation needed because IPs are real VPC IPs.

0:50 – 1:10  UNDERSTAND NetworkPolicy:
             By default: all pods can talk to all pods (no isolation).
             NetworkPolicy: K8s firewall rules for pod-to-pod traffic.
             Example: Only allow frontend pods to talk to backend pods.
                      Block backend pods from talking to external internet.

1:10 – 1:30  RUBBER DUCK: "AWS VPC CNI is different from Flannel because..."

### DAY 32 — "K8s STORAGE — PV, PVC, STORAGECLASS"
```
0:00 – 0:40  UNDERSTAND THE HIERARCHY:

             StorageClass: Defines HOW storage is provisioned
             → AWS EBS StorageClass: creates EBS volumes on demand
             → AWS EFS StorageClass: creates EFS (shared, multi-pod) storage

             PersistentVolume (PV): Actual storage resource (could be an EBS volume)
             → Created manually (static) or automatically by StorageClass (dynamic)

             PersistentVolumeClaim (PVC): Pod's REQUEST for storage
             → Pod says "I need 10Gi of storage"
             → K8s finds a matching PV (or creates one via StorageClass)
             → PVC binds to PV
             → Pod mounts the PVC

             Access Modes:
             ReadWriteOnce (RWO): Only ONE node can mount (EBS)
             ReadOnlyMany (ROX): Many nodes can read
             ReadWriteMany (RWX): Many nodes can read+write (EFS)

             Important: EBS is RWO — one pod on one node.
             EFS is RWX — multiple pods across nodes can share it.
             Use EFS for shared config files, shared media storage.

0:40 – 1:00  PRACTICE:
             $ kubectl get pv                         → list persistent volumes
             $ kubectl get pvc -n [namespace]         → list claims
             $ kubectl describe pvc [name]            → see binding status

1:00 – 1:30  RUBBER DUCK: "The flow from StorageClass → PV → PVC → Pod is..."
             "I would use EFS instead of EBS when..."
```

### DAY 33 — "K8s ADVANCED SCHEDULING"
```
0:00 – 0:50  UNDERSTAND SCHEDULING CONTROLS:

             nodeSelector: Simple — run pod ONLY on nodes with this label
             spec:
               nodeSelector:
                 disk: ssd

             Node Affinity: More flexible nodeSelector with required/preferred
             → requiredDuringScheduling: MUST run here (hard rule)
             → preferredDuringScheduling: TRY to run here (soft rule)

             Taints and Tolerations: Keep pods AWAY from nodes (unless they tolerate)
             → Use case: dedicated nodes for system pods, GPU nodes, spot instances
             → Taint a node: kubectl taint nodes node1 key=value:NoSchedule
             → Pods must have matching toleration to schedule there

             PodAffinity/Anti-Affinity: Place pods relative to OTHER pods
             → Anti-affinity: spread replicas across nodes (HA)
               "Don't run my pods on a node that already has another copy of me"
             → Affinity: co-locate pods on same node (performance)

             Interview question: "How do you ensure HA for your application pods?"
             Answer: "I use PodAntiAffinity to ensure replicas spread across nodes.
             This way, if a node fails, not all replicas go down simultaneously."

0:50 – 1:10  UNDERSTAND POD DISRUPTION BUDGET (PDB):
             → Ensures minimum number of pods stay available during voluntary disruptions
             → Example: During node drain/upgrade, keep at least 2 out of 3 replicas running
             → minAvailable: 2 (keep at least 2)
             → maxUnavailable: 1 (allow max 1 down at a time)

1:10 – 1:30  RUBBER DUCK: "Taints and tolerations vs node affinity — difference is..."
             "I use PDB to prevent..."
```

### DAY 34 — "K8s CLUSTER OPERATIONS — UPGRADES + BACKUP"
```
0:00 – 0:40  CLUSTER UPGRADE PROCESS (Critical for CKA + interviews):

             Control plane upgrade first, then worker nodes.
             Never skip versions — upgrade one minor version at a time.

             EKS Upgrade Process:
             Step 1: Update EKS control plane version via AWS console/CLI
             Step 2: Update kube-proxy, CoreDNS, VPC CNI add-ons

## WEEK 9: "THE SYSTEM DESIGN FRAMEWORK"

### DAY 61 — "HOW TO APPROACH ANY SYSTEM DESIGN QUESTION"

```
0:00 – 0:20  THE FRAMEWORK — use this for EVERY system design question:

             Step 1: CLARIFY (3–5 minutes)
             Ask before you design. Never assume.
             → How many users? DAU? Requests/second?
             → Read-heavy or write-heavy?
             → Any specific latency requirements?
             → Global or single region?
             → Any existing infrastructure to integrate with?

             Step 2: ESTIMATE SCALE (2–3 minutes)
             → Example: 10M DAU, 100 requests/user/day = 1B requests/day
               = 1B / 86400 seconds = ~11,600 RPS
             → Storage: 1M writes/day × 1KB average = 1GB/day = 365GB/year
             → This tells you: do I need sharding? caching? CDN?

             Step 3: HIGH LEVEL DESIGN (5–10 minutes)
             Draw boxes: Client → CDN → Load Balancer → App Servers → Cache → DB
             Explain the happy path first.

             Step 4: DEEP DIVE (10–15 minutes)
             Interviewer will ask you to go deeper on one component.
             → DB design (schema, sharding strategy)
             → Caching layer (what to cache, eviction policy)
             → Scaling bottlenecks (where does it break? how do you fix it?)

             Step 5: TRADE-OFFS (3–5 minutes)
             → What are you sacrificing for this design?
             → Consistency vs availability? Cost vs performance?
             → What would you do differently with more time?

0:20 – 0:50  MEMORIZE THESE NUMBERS (every system design uses them):
             1 million requests/day ≈ 12 requests/second
             1 billion requests/day ≈ 12,000 requests/second
             1 KB × 1M = 1 GB
             MySQL handles ~1,000–2,000 writes/second comfortably
             Redis handles ~100,000 operations/second
             Kafka handles millions of messages/second

0:50 – 1:30  PRACTICE: Apply the framework to this question:
             "Design a URL shortener"
